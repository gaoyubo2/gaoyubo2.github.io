论文精读-Understanding RDMA Microarchitecture Resources for Performance Isolation

疼了也不哭2023-12-272023-12-27

Abstract

近年来，RDMA 在云中得到广泛采用，加速第一方工作负载并通过释放 CPU 周期来节省成本。现在，云提供商正在努力在通用来宾虚拟机中支持 RDMA，以使第三方工作负载受益。为此，云提供商必须提供强大的性能隔离，以便一个租户的 RDMA 工作负载不会对另一租户的 RDMA 性能产生不利影响。尽管在公有云中的网络性能隔离方面做出了很多努力，但我们发现 RDMA 由于其复杂的 NIC 微架构资源（例如 NIC 缓存）而带来了独特的挑战。

在本文中，我们旨在系统地了解RNIC微架构资源对性能隔离的影响。 我们提出了一个模型来表示 RDMA 操作如何使用 RNIC 资源。 使用此模型，我们开发了一个测试套件来评估 RDMA 性能隔离解决方案。 我们的测试套件可以打破各种场景下的所有现有解决方案。 我们的结果得到了最大的 RDMA NIC 供应商之一的认可和复制。 最后，根据测试结果，我们总结了设计未来 RDMA 性能隔离解决方案的新见解。

1 Introduction

在共享计算基础设施上复用来自不同租户的工作负载开启了现代云计算时代。全球云基础设施收入已超过4000亿美元，预计未来十年将增长至1万亿美元左右[7]。

众所周知，租户的工作负载不同，共享计算资源可能会导致不可预测的应用程序性能干扰 [12, 18, 66] 和隐私泄露 [32, 39]。这推动了大量研究关注云中的性能隔离，特别是对于具有严格服务级别目标的性能关键型应用程序[11,12,18,41,63,66,70]。实践中的最新技术也取得了显着的进步：CPU 供应商甚至实现了硬件机制来控制和隔离 CPU 缓存 [20]。随着时间的推移，通过共享资源的侧通道正在被修复[39]。

在本文中，我们访问一种特定的硬件设备：RDMA NIC (RNIC)。 RDMA 将网络堆栈从操作系统内核卸载到 NIC 硬件，以提供高吞吐量和超低处理延迟，并且 CPU 开销接近于零。 RDMA 已大规模部署在数据中心中，以提高性能并释放 CPU 内核以用于存储和 ML 等第一方工作负载 [14,17,38,51]。 现在，云提供商正在努力在通用来宾虚拟机中支持 RDMA，以使第三方工作负载受益。 为此，云提供商必须为共享同一 RNIC 的租户提供强大的性能隔离。

人们已经做出了许多努力来提高公共云中的网络性能隔离，特别关注带宽和数据包处理能力[3,15,16,25,34,62,64]。 然而，由于其独特且复杂的NIC微架构资源（例如NIC缓存和处理单元），RDMA带来了新的挑战。 学术界已经知道它们的存在和对性能的影响 [29, 33]。 为了避免性能异常，开发人员精心设计 RDMA 系统，以避免耗尽这些微架构资源 [5,9,10,27,30,50,61]。 我们的研究是从不同的角度进行的：我们从公有云供应商的角度来研究这些微架构资源如何影响 RDMA 性能隔离。 云供应商不了解和控制租户的RDMA应用程序，租户可以以任意方式使用RNIC微架构资源。

为了证明 RNIC 微架构资源对性能隔离的重大影响，我们测试了最先进的方法：使用具有分离硬件流量类别 (HW TC) 的 SR-IOV。 SR-IOV 和 HW TC 都是商用 RNIC 上可用的硬件机制。 HW TC 在 RNIC 中利用多个硬件队列（通常是 8 个队列）。我们可以分配每个租户应用程序使用一个队列。 我们使用 ib_write_bw（Perftest [56] 中的标准 RDMA 带宽测试工具）在两个虚拟机之间运行一个受害者流量。 每个虚拟机位于不同的服务器上，这两个服务器都配备了 100 Gbps NVIDIA ConnectX-5 RNIC。 图 1 显示了带宽。 带宽测试达到80Gbps。 我们在每台服务器上启动一个虚拟机来代表攻击者（即有错误或恶意的租户应用程序），并启用性能隔离以向受害者和攻击者授予总带宽的一半。 受害者流量减少至 50 Gbps，这是预期的。 然而，当我们开始精心设计的只有 1 Gbps 的攻击者流量来故意耗尽 RNIC 微架构资源之一时，受害者立即下降到 2 Gbps，违反了性能隔离保证（即为受害者保证 50 Gbps 的网络带宽） 。

我们开发了一组实验来研究不同类型的 RDMA 操作如何使用 RNIC 微架构资源。 我们的实验得出了一些有趣的发现，包括：（1）异常或错误处理会暂停 RNIC 的pipelines并导致其他租户的性能急剧下降。 (2) 控制 verbs导致缓存未命中，严重增加并损害其他租户的性能。 (3) 数据 verbs耗尽不同类型的微架构资源并违反性能隔离。 据我们所知，我们是第一个系统研究所有类型的操作和异常对 RDMA 微架构资源消耗的影响的。

我们利用这些发现创建 RDMA 操作模型来描述 RDMA 操作动作与消耗的微架构资源之间的关系。 我们的模型使我们能够了解如何耗尽每个 RNIC 资源。 使用该操作模型，我们创建了第一个测试套件Husky，以系统地测试和评估RNIC性能隔离解决方案。 不幸的是，在商用 RNIC 上运行我们的测试套件揭示了坏消息：目前没有可以提供 RNIC 性能隔离的解决方案。 我们已经向三大 RNIC 供应商 NVIDIA、Chelsio 和 Intel 报告了我们的所有发现。 我们的结果得到了最大 RDMA NIC 制造商之一 NVIDIA 的完全重现和认可。 最后，我们提出了关于如何构建未来性能隔离解决方案的新见解。 我们希望这些见解能够有益于未来的 RNIC 设计和 RDMA 软件开发。

本文做出以下贡献：

  • 我们确定了 RDMA 操作和 RNIC 微架构资源之间的多种交互，包括错误处理和控制操作的先前未知的影响。
  • 我们引入第一个RDMA操作模型来描述RNIC 微架构资源在操作动作（标准 RDMA 编程 API）中如何消耗，以及为什么这些微架构资源会影响性能隔离。
  • 我们构建了第一个测试套件来系统地测试和评估 RNIC 性能隔离解决方案。 我们表明现有的性能隔离解决方案都无法通过我们的测试套件。 Husky 测试套件可从 GitHub - host-bench/husky: Benchmark Suite for RDMA Performance Isolation 获取。

这项工作表明，在公有云中为 RDMA 提供性能隔离比人们想象的要困难得多。 未来的RDMA性能隔离解决方案必须有更高的标准：应仔细考虑RNIC微架构资源并通过系统基准进行评估。

2 Background and Motivation

我们首先介绍公有云中网络性能隔离的背景知识。 然后我们介绍了RDMA并讨论了RDMA网络性能隔离带来的新挑战。

2.1 Network Performance Isolation in the Public Cloud

云中的租户主要造成两类网络资源的争用。 第一个最明显的问题是网络结构中的带宽。 为了缓解租户之间的带宽争用，第一项工作 [58,60,62] 静态地限制每个租户的带宽。 另一条工作线[1,3,4,6,16,24,25,37,58,59,68]为每个租户提供最低带宽保证，并允许租户使用空闲带宽容量。 第二种类型的资源是终端主机处的数据包处理资源。 每个数据包的处理成本取决于许多因素，例如缓存未命中和要执行的操作。 最近，PicNIC [34] 为此类软件数据包处理提供了隔离。 人们还利用专门的硬件来实现相同的目标[64]。

值得注意的是，网络性能隔离与网络虚拟化有很大不同。 网络虚拟化编排网络资源，为每个租户提供独立网络的错觉。 租户不应影响另一个租户的网络连接。 网络虚拟化的目标是实现低开销[19,31,57]。 相比之下，网络性能隔离侧重于如何管理资源争用，以确保租户能够获得有保证的性能。

2.2 RDMA Overview

RDMA 允许 NIC 直接在线路和应用程序内存之间传输数据。 网络协议在 NIC 中实现。 图 2 展示了 RDMA 工作流程的概述。 它将标准 RDMA 编程接口（也成为verbs）分为两类：控制verbs和数据verbs。 应用程序首先需要调用几个控制动作来分配必要的对象，例如队列对（QP）和完成队列（CQ），以建立可靠连接（RC）、不可靠连接（UC）或不可靠数据报（ UD）传输端点。 然后应用程序需要注册一个内存区域（MR）。 这种注册实质上是固定主机DRAM中的内存并获得从虚拟地址到物理地址的映射，这使得RNIC能够直接读取或写入该内存区域。 所有这些控制verbs均按以下过程进行处理：RDMA 的用户空间库和内核驱动程序处理动作请求，生成请求命令，将命令放入协商命令队列中，然后按响 RNIC 的门铃（例如，内存映射寄存器）。 RNIC 从命令队列中取出命令，对其进行处理，并将响应推回到队列中。 然后驱动程序处理响应并将对象返回给应用程序。

经过上述初始化后，应用程序就可以开始本地和远程内存之间的数据传输了。 应用程序可以使用多种类型的操作，例如 SEND/RECV、WRITE、READ 和 ATOMIC。 我们将这些操作命名为数据verbs。 为了发出数据verbs，应用程序通常将请求发送到其发送队列，并通过用户空间库敲响 RNIC 的门铃。 然后RNIC解析该请求，从主机存储器读取数据，将数据分段成数据包，并传输数据包。 此过程绕过内核。 不同类型的请求的处理存在一定的差异。 例如，对于 SEND/RECV 消息，接收方应在发送方发出 SEND 请求之前发布足够的 RECV 请求。 否则，传入的 SEND 请求可能会被丢弃或需要重传，因为接收方 RNIC 缺乏接收请求来处理它们，这称为接收未就绪 (RNR) 错误。 对于向/从远程端写入/读取数据或执行原子操作，发送方应指定正确的远程虚拟地址和内存键。 无效地址或错误密钥将触发内存保护错误并导致 QP 转换为错误状态。

2.3 Why RDMA Performance Isolation is Hard?

如上所示，RDMA将许多主机网络功能卸载到RNIC，RNIC具有许多不可见的硬件组件，并且每个组件都可能单独成为性能瓶颈。 图 3 显示了商用 RNIC 的硬件组件。 我们根据 NVIDIA 的公开文档绘制此图 [44,46,48]。 除了数据包缓冲区（TX/RX Buffer）之外，RNIC 还具有多个处理单元（PU）和多种类型的内部缓存。 每个内部缓存用于存储特定类型的元数据。 例如，在 NVIDIA RNIC 中，互连上下文内存 (ICM) 缓存存储 QP 上下文； 内存转换表（MTT）和内存保护表（MPT）存储内存地址转换和保护信息的条目； 工作队列条目 (WQE) 缓存存储预取的发送 WQE 和发布的接收 WQE。 由于这些缓存源自设计需求，其他 RNIC 也包含类似的组件。 基于 CPU 硬件的类比，我们将这些 RNIC 硬件组件命名为微架构资源。 CPU 的设计符合标准指令集架构（例如 ARM、x86），但 CPU 设计人员可以做出微架构级别的决策，例如缓存的级别和缓存大小。 RNIC 是相似的，因为 RNIC 供应商必须为 RDMA 应用程序开发人员提供相同的编程接口，但供应商可以决定这些微架构级别的细节，例如 RNIC 缓存。

之前的许多工作已经确定了这些微架构资源对 RDMA 应用程序性能的一些影响。 例如，[5,29,50]发现RNIC缓存QP上下文。 QP 上下文缓存未命中可能会触发 RNIC 的额外 PCIe 往返行程，以从主机 DRAM 获取上下文，从而降低应用程序性能。 例如，200 个连接可能会导致 NVIDIA ConnectX-3 NIC 上的请求率下降 90% [5]。 然而，这些努力是从应用程序开发人员的角度研究微架构资源。 在性能下降之后，他们识别瓶颈资源，寻求更有效的方法来使用数据动作，并相应地修改他们的应用程序。

然而，在公共云中，云供应商无法控制租户的应用程序。 因此，租户可以按照自己的意愿甚至恶意地使用 RNIC 的微架构资源。 因此，从云供应商的角度来看，我们需要了解大多数（如果不是全部）RDMA 动作的微架构资源消耗，而不仅仅是常见的数据动作。 只有掌握了这些知识，我们才能正确地将 RNIC 的微架构资源分配给不同的租户，以提供可预测的性能。

3 RNIC Microarchitecture Resources

在本节中，我们将介绍我们目前所知的所有 RNIC 微架构资源的研究。 先前的工作已经确定了几种特定形式的资源争用。 但我们的目标是系统地研究所有可能类型的资源争用。 对于每个微架构资源，我们研究三类 RDMA 操作如何消耗它：(1) 为应用程序分配对象的控制verbs（例如 ibv_create_qp），(2) 启动数据传输的数据verbs（例如，ibv_post_send），以及 (3)异常处理操作，处理异常或错误（例如，RNR错误）。 由于篇幅限制，我们首先介绍一些对 RNIC 性能隔离具有重大影响的关键发现。 之后，我们总结了其他几个发现。 我们在附录 B 中详细分析了 NVIDIA 对这些调查结果的回应。

3.1 Methodology

我们的研究结果围绕如何通过标准 RDMA 编程 API verbs 接口 [21] 耗尽 RNIC 微架构资源。 对于每个关键发现，我们都通过具体设置进行演示，其中包括受害者工作负载和攻击者工作负载。 尽管我们使用术语“攻击者”，但攻击者租户不会通过漏洞对其他租户进行未经授权的访问。 相反，攻击者只是发出标准 RDMA verbs的普通 RDMA 应用程序。 每个租户有一个客户端和一台服务器。 受害者和攻击者的客户端位于同一台物理机上，共享同一个RNIC。 受害者和攻击者的服务器位于不同的物理服务器上。 在测量过程中，我们没有启用任何隔离机制。 我们将在第 5 节中研究现有的性能隔离解决方案。

我们重点关注受害者和攻击者之间通过耗尽微架构资源而造成的性能干扰。 我们首先仅运行受害者，以使链路带宽容量（每秒位数）或 RNIC 的最大请求速率（每秒请求数）达到饱和。 然后我们启动攻击者并测量受害者和攻击者的两个指标。 如果不存在微架构资源争用，则两个租户的性能指标之和应符合规范中 RNIC 的限制。 现代 RNIC 指定其带宽容量和请求速率限制。 如果两个租户的性能指标之和低于两个指定的限制，我们将其归因于微架构资源的争用。 例如，假设没有攻击者，受害者可以达到 100 Gbps。 然而，对于 X Gbps 攻击者，受害者会降低至 Y Gbps，并且 X +Y < 100。我们还假设总请求速率低于 RNIC 规范。 在这种情况下，我们得出结论，某些微架构资源出现瓶颈。 除非另有说明，流量均使用 RC 连接。

我们测试了四种类型的 100 Gbps RNIC：NVIDIA ConnectX-5 EN 和 ConnectX-6 Dx、Chelsio T62100-LPCR 和 Intel E810。 NVIDIA NIC 运行 RoCE，Chelsio NIC 运行 iWARP。 Intel E810 同时支持 RoCE 和 iWARP，但我们目前仅测试其 RoCE 实现。 RoCE 和 iWARP 是在基于以太网的网络上运行 RDMA 的两种标准方法。 我们的测试床由两台服务器组成，每台服务器都配备一个 RNIC，两个 RNIC 通过 100 Gbps 交换机连接。 对于 NVIDIA RNIC，我们可以通过其网络适配器管理工具 NEO-Host [44] 访问其硬件计数器，例如缓存未命中计数器。 这些硬件计数器使我们能够查明哪个资源被超额认购。 例如，当 ICM 缓存未命中计数器随着特定应用程序工作负载而快速增加时，我们了解到该工作负载大量使用此缓存，导致其超额订阅。 由于其他 RNIC 不公开此类计数器，因此我们根据其他 RNIC 的端到端性能指标（例如带宽）对其进行实验。

3.2 NIC Caches

我们知道RNIC至少有三种类型的缓存，如图3所示。RNIC在这些缓存中存储多种类型的元数据以加速请求处理，例如ICM缓存中的QP上下文。 先前的工作已经识别出一些由具有特定模式的数据动词引起的 RNIC 缓存争用问题。 例如，同时在许多 RC QP 上传输小消息以及对大量内存区域的随机访问可能会导致某些类型的严重缓存未命中（例如，ICM 和MTT/MPT）[29, 53]。 ScaleRPC [5] 发现这种可扩展性问题可以使 WRITE 请求率降低 90%。 除了这些众所周知的问题之外，我们还观察到一种新的、甚至更严重的耗尽缓存的方法：

主要发现#1：控制动词可能导致过多的缓存未命中和性能急剧下降。 控制verbs（例如 ibv_reg_mr）用于创建和销毁 MR 和 QP 等对象，数据verbs将使用这些对象来传输数据。 据我们所知，还没有关于控制verbs如何消耗 RNIC 微架构资源的研究。 我们发现控制verbs很容易触发过多的缓存未命中，从而降低带宽和请求率。

我们通过 NVIDIA ConnectX-5 RNIC 上的简单实验证明了这一发现。 我们让受害租户使用 6 个core，每个core 16 个连接，发出 512B WRITE 请求，以耗尽 RNIC 的带宽容量（即 100 Gbps）。 结果如表1所示。 受害者可以达到 96.6 Gbps，MTT 缓存未命中率为 17.2%。 在这样的缓存未命中率下，受害者仍然可以实现线速率，因为 QP 复用和 RNIC 管道设计可以在一定程度上掩盖缓存未命中的开销。 我们让单线程攻击者在受害者的发送方使用 ibv_reg_mr （每秒约 5K 注册）继续注册内存区域 (MR)。 在这种情况下，受害者的带宽几乎不受影响，保持在 95.9 Gbps，丢失率略有上升至 22.9%。 然而，如果攻击者不断注销MR，我们可以看到对受害者的显着影响：缓存未命中率增加到49.1%，带宽下降到48 Gbps。 如此高的缓存未命中率下的开销变得很大并且不再能够被RNIC处理管道屏蔽。 值得注意的是，攻击者不需要发出任何数据动词，因此攻击者根本不消耗网络带宽或请求速率。 幸运的是，我们观察到这种干扰在接收器侧可以忽略不计。

与数据verbs相比，我们发现控制verbs更容易造成性能干扰。 溢出的缓存资源，我们需要启动足够的动态数据verbs并强制它们随机访问大量对象（例如，MR）。 例如，在 NVIDIA ConnectX-5 RNIC 上，我们发现需要 6 个线程才能以 96 个 QP 访问超过 18K MR，从而导致严重的 MTT 缓存未命中，导致带宽下降 40.1%。 我们相信，由于 RNIC 供应商不断增加片上缓存资源，由数据verbs引起的缓存未命中将变得不那么严重。 相反，控制verbs通过其特殊语义影响缓存资源，而不是简单地消耗它们，因此控制verbs的影响可能很难减轻。 例如，我们推测MR注销可能会使整个MTT/MPT缓存失效，以避免访问过时的MR。 这会导致访问其他 MR 时缓存未命中。 我们还在 Chelsio 和 Intel NIC 上进行了相同的实验，我们观察到了类似的结果。

3.3 Processing Units

RNIC 有多个处理单元 (PU) 来处理动词verbs。 由于缺乏公共可用的计数器来监控 PU 的状态，我们使用请求率作为衡量不同动词如何消耗 PU 的指标。 我们总结了以下两个主要发现：

主要发现#2：不同数据verbs之间的性能干扰取决于verbs的复杂性。 不同的数据verbs有不同的复杂性。 简单的verbs，如发送和读取，仅在机器之间复制数据。 复杂动词（例如 fetch_and_add）以原子方式将 64 位值添加到远程地址的内存中。 此操作利用 PCIe 功能（例如，读取-修改-写入事务），并且还可能获取目标地址的锁定。 这些复杂的verbs消耗更多的PU资源，导致请求率较低[29]。 我们的新发现是，资源消耗的这种差异还可以通过资源耗尽开辟一条新的性能干扰途径：当受害者与大量使用复杂verbs的攻击者共处一地时，受害者的性能可能会受到严重影响。

为了理解这种效果，我们首先测量与其他数据verbs竞争时的数据verbs请求率。 我们从 NVIDIA 100 Gbps ConnectX-5 RNIC 开始。 我们为每个测试设置两个工作负载，每个工作负载在 8 个专用 CPU 核心上运行 8 个 QP，以使 RNIC 的速率达到饱和。 为了避免 RNIC 严重的缓存未命中，我们总共只使用 128 个 QP 和 16 个 MR。 我们在所有 PU 测试中观察到缓存未命中率低于 1%。 为了避免达到带宽容量限制，我们使用 8B 作为所有数据verbs的请求大小。 我们首先使用特定类型的数据verbs设置一个工作负载（受害者），然后使用不同类型的数据verbs设置攻击者工作负载。 我们在图 4 中显示了他们的请求率结果。

我们的第一个结论是，除了 ATOMIC 操作 [29] 之外，READ 操作也比 SEND/RECV 和 WRITE 更昂贵。 当它们单独运行时（作为受害者流量），FAA 和 CAS 仅分别达到 5.2 Mrps 和 4.8 Mrps。 READ 达到约 60 Mrps。 SEND和WRITE可以达到90Mrps以上。

第二个也是更重要的一点是，数据verbs操作的不同组合之间的争用行为可能会有所不同。 例如，当受害者单独运行 READ 工作负载时，它可以达到 60 Mrps。 如果攻击者运行 CAS 工作负载，受害者的请求率立即降至 3 Mrps。 如果攻击者运行 READ 工作负载，受害者的请求率只会下降到 30 Mrps。 这意味着复杂verbs（例如 CAS）可能会消耗更多资源，并影响其他并置动词工作负载。 我们要强调的一个非直观行为是，如果攻击者在此设置下运行 SEND 或 WRITE 工作负载，则运行 FAA 或 CAS 的受害者的请求率实际上会增加。

我们还在100 Gbps Chelsio T62100-LP-CR RNIC上进行了类似的测试，结果如图5所示。该iWARP RNIC不支持ATOMIC操作。 我们观察到，与 RoCE RNIC 相比，iWARP RNIC 对所有类型数据verbs的请求率较低，这与之前的研究结果相匹配 [8,49,71]。 我们发现 Chelsio 的 RNIC 上数据verbs之间的争用也各不相同。 例如，具有 WRITE 工作负载的受害者可以在没有干扰的情况下达到 4.76 Mrps。 攻击者可以导致受害者的 SEND 工作负载请求率下降 55.0%，READ 工作负载下降 73.1%。 具体模式与 NVIDIA RNIC 不同，但这个结果仍然证明了我们的关键发现：不同数据verbs的 PU 开销有所不同。

主要发现#3：错误处理可能会导致 RNIC 处理单元停顿并挂起所有应用程序。 RNIC 需要处理几种类型的错误，包括传输超时（响应方未发送 ACK 或 NACK）、接收未就绪 (RNR) 错误（响应方没有足够的接收请求来接收到达的发送请求）、本地或 远程保护错误（发布的请求未引用有效的本地或远程内存区域）和本地操作错误（操作码在错误类型的 QP 上操作）。 处理这些错误需要 RNIC 处理单元的资源，并且 RNIC 处理某些错误的成本可能很高。

在 NVIDIA ConnectX-5 和 ConnectX-6 RNIC 上，我们发现处理 RNR 错误可能会完全停止 RNIC 处理单元。 对于受害者，我们使用 Perftest [56] 在单个 QP 上保留 128 个未完成的 64KB 写入请求，以使带宽容量饱和。 对于攻击者来说，我们只使用单个 QP（即表中的 SEND 应用程序）来仅保留一个正在进行的 4KB SEND 请求，以消耗少量带宽。 如表 2 所示，如果 SEND 应用程序正常生成流量（例如，响应者发出足够的接收请求），它会消耗 4 Gbps 带宽，而受害者的带宽仅下降约 3.5 Gbps。 然而，当SEND应用程序触发RNR错误时（例如，响应方没有发布任何接收请求），SEND应用程序和受害者都会停止。 我们在两个方向上测试此 RNR 误差并看到相同的结果。 原因是RNR接收方的RNIC被停顿，RNIC甚至无法处理ACK包。 因此，即使受害者向相反方向发送流量，他们也会陷入停滞。

我们使用 Intel 和 Chelsio NIC 进行相同的实验。 我们观察到，在 Intel E810 的数据传输过程中，受害者的 QP 连接也会意外终止。 幸运的是，我们没有看到 Chelsio T62100-LP-CR 出现此类 RNR 问题。 我们最好的猜测是，iWARP 是在 TCP 之上设计的，旨在在有损结构上运行，因此它可能具有更有效的错误处理机制。

3.4 PCIe Bandwidth

RNIC 连接到 PCIe 控制器，并使用 PCIe 通道从 CPU 传输数据或向 CPU 传输数据。 之前的几项工作已经研究了 PCIe 对网络堆栈的影响 [29,34,52]。 基于现有的 PCIe 模型，我们进一步研究 RDMA verbs如何消耗甚至耗尽 PCIe 带宽。 之前的工作已经确定了 RDMA 环回流量如何耗尽 PCIe 带宽 [26, 33]。 因此，我们关注正常的 RDMA TX 和 RX 流量。 为了传输 RDMA 消息，PCIe 引入了以下类型的额外字节：(1) 用于敲响 RNIC 上门铃的 MMIO（64B，取决于缓存行大小），(2) 工作队列元素 (WQE)（36B 或 64B），（3）PCIe 协议开销（例如，TLP 标头），以及（4）由缓存未命中触发的额外 PCIe 操作。 我们对 PCIe 带宽的主要观察是：

主要发现#4：只有当请求大小在特定范围内时，PCIe 带宽才会成为瓶颈。 我们只需要一个租户来证明这一重要发现。 我们在 NVIDIA 100 Gbps ConnectX-5 RNIC 上运行实验。 PCIe 带宽容量为 128 Gbps (PCIe Gen 3.0 x16)。 我们在 6 个核心上使用 96 个 QP 来饱和 PCIe TX 带宽。 每个 QP 保留 256 个未完成的 WRITE 请求。 我们改变请求大小，并通过读取 RNIC 计数器来收集 NIC 和 PCIe 带宽消耗。 结果如图 6 所示。我们首先观察到，当有效负载较小时，商品 RNIC 可以通过在 WQE 中嵌入小消息来减轻 WQE 开销。 如绿色矩形所示，当请求大小小于28B时，增加请求大小不会导致更多的PCIe带宽消耗，因为有效负载与WQE嵌入在相同的MMIO操作中。

我们的第二个观察结果是，只有当请求的有效负载大小在特定范围内时，PCIe TX 带宽才可能成为瓶颈。 原因是短请求在耗尽 PCIe 带宽之前首先受到请求速率的限制，而大请求始终受到 RNIC 带宽容量的限制。 我们通过理论 PCIe 消耗模型证实了这一观察结果，并提出了两个具体示例。 我们假设网络 MTU 为 4096B，每个 PCIe 事务的最大有效负载为 128B（最大化 PCIe 开销的最差设置）。 TLP 开销取决于实现[52]，我们假设它为 20B，这是 PCIe 3.0 设备的典型大小。 传输 29 字节的消息将消耗最多 127 个网络字节和至少 189 个 PCIe 字节 [29, 69]。 因此，要使链路带宽（100 Gbps）饱和，我们至少需要148.8 Gbps的PCIe带宽，这远大于PCIe 3.0x16的容量。 附录A包括详细的计算。 我们的测量表明，实际消耗可能更高，如图 6 所示。PCIe RX 带宽（即 RNIC 到主机）的消耗模型与 TX 的消耗模型类似。 此外，过多的缓存未命中还可能由于大量 PCIe 读取来获取元数据而导致 PCIe 带宽消耗较高。 然而，在大多数情况下，大量的缓存未命中首先会减慢RNIC的执行速度（例如，引入额外的延迟），因此PCIe带宽的消耗较少。 在我们对缓存未命中的测量中，我们没有观察到 PCIe TX 带宽耗尽的情况。

理论模型和我们的实验结果都表明 PCIe 带宽可能成为瓶颈，但仅限于特定的请求大小范围。

3.5 Other findings

我们还有其他一些有趣的发现。 限于篇幅，我们在这里只简单介绍一下。 然而，我们确实使用这些发现来指导我们在第 4 节中的测试套件设计。

其他发现#1：数据verbs争夺不同的 RNIC 缓存。 我们使用不同的数据verbs进行可扩展性测试，并观察不同类型的缓存争用。 例如，大量发出READ和WRITE的RC QP将主要导致ICM缓存未命中。 大量发出 SEND/RECV 请求的 UD QP 或许多发出 ATOMIC 请求的 RC QP 可能会导致严重的 RECV WQE 缓存未命中。 这一观察结果表明，数据verbs以不同的方式争夺缓存，类似于 RNIC PU 上的争用。

其他发现 #2：跨多个对象（QP、CQ、MR）的大范围访问会导致 ICM 缓存未命中。 可扩展性问题已经得到了充分研究，但我们的测量揭示了新的观察结果。 除了QP和MR之外，ICM缓存中还存储了完成队列（CQ）的上下文。 因此，访问大量 CQ 也会触发严重的 ICM 缓存未命中。 此外，分配大量这些对象并不一定会导致严重的 ICM 缓存未命中。 跨对象的大范围访问（即较差的局部性）是触发严重 ICM 缓存未命中和性能下降的关键。

其他发现#3：控制verbs的影响受到其核心参与的限制。 我们观察到所有控制verbs首先由内核驱动程序处理，从而导致昂贵的上下文切换。 这些控制verbs的执行速率通常由内核而不是 RNIC 处理来限制。 因此，控制verbs对耗尽 RNIC PU 的影响有限。 然而，正如我们的主要发现#1 所示，它们仍然会导致严重的性能干扰，并通过触发严重的缓存未命中来影响其他应用程序。

我们在图 7 所示的 RDMA 操作模型中总结了我们的发现。该模型描述了verbs操作大量消耗的微架构资源。 请注意，verbs操作还可以使用我们的实验未捕获的其他微架构资源。 这是因为这些资源的使用率较低，不会导致资源争用。 该模型是定性的：我们不会尝试了解确切的资源使用情况，因为我们无法了解专有的 RNIC 硬件。 例如，我们知道某种流量模式可以触发某种类型的缓存未命中，但我们不知道缓存的总大小或操作消耗了多少缓存。 即便如此，我们证明这个模型足够强大，足以让我们创建第一个 RNIC 性能隔离测试套件，并且它可以捕获可以破坏现有性能隔离解决方案的各种工作负载。

4 The Husky Test Suite

在了解不同的 RDMA 操作如何使用这些微架构资源后，我们可以设计一个测试套件来评估性能隔离解决方案。 我们的目标如下：给定 RNIC 硬件和性能隔离解决方案，我们希望为攻击者和受害者找到一组可以打破性能隔离的工作负载组合。 我们需要检查不同的受害者工作负载的完整性，因为不同的受害者工作负载对不同微架构资源的耗尽很敏感。

我们的测试套件必须是通用的：我们将使用它来测试不同 RNIC 上的各种 RNIC 性能隔离解决方案。 这意味着我们不能依赖特定供应商的工具和功能，例如 Mellanox Neo-Host [44]。 另外，不同的RNIC具有不同数量的微架构资源。 现有的性能隔离解决方案可能只能减轻对特定资源的争用。

为此，我们构建了Husky来系统地测试和评估RNIC性能隔离解决方案。 Husky针对四种类型的资源：网卡带宽、PCIe带宽、网卡PU和网卡缓存。 对于每种类型的资源，我们设计具有不同类型行为（例如控制verbs）的合成工作负载以耗尽该资源。 更具体地说，我们使用不同的操作码（例如，WRITE）通过长消息耗尽 NIC BW； 我们通过环回流量和特定消息模式耗尽 PCIe 带宽（来自关键发现 #4）； 我们用昂贵的数据verbs（关键发现#2）、小消息或错误处理行为（关键发现#3）耗尽NIC PU； 我们通过密集的控制verbs（关键发现#1）和广泛的数据verbs访问来耗尽不同类型的 RNIC 缓存。 我们改变参数（例如，连接类型)的一些综合工作负载更具包容性。 Husky 总共包括 52 个攻击者合成工作负载（6 个用于 NIC BW、4 个用于 PCIe BW、14 个用于 NIC PU、28 个用于 NIC 缓存）和 20 个合成受害者工作负载。 许多攻击者工作负载无法使用现有 RDMA 流量引擎直接生成。 因此，我们扩展了 Collie [33] 的流量引擎（据我们所知是最灵活的引擎）来生成这些合成 RDMA 流量，包括灵活的控制动词工作负载和错误处理工作负载。

Husky 的框架还可以轻松地允许运行真实应用程序作为额外的受害者工作负载。 Husky 目前包含两个实际应用程序，包括 OSU 基准测试 [54] 和基于 eRPC 的 Masstree 键值存储 [27, 40]。 OSU 基准测试包含 allreduce 和 allgather 等工作负载。 请注意，我们可以将任何 RDMA 应用程序集成到 Husky 中。 我们从测试套件中详尽地测试了所有（受害者、攻击者）工作负载对。

一个关键问题是如何定义违反性能隔离的行为。 我们对违规的定义取决于具体的隔离解决方案。 Husky 使用用户指定的verbs来计算启用隔离时的预期性能结果。 Husky 将实际性能与预期性能进行比较，以识别违规行为。 例如，现有的性能隔离方案大多只提供带宽保证。 因此，这些隔离解决方案的预期性能是有保证的带宽 Bg。 我们假设应用程序单独运行时会消耗 Ba 的带宽。 在任何攻击者工作负载下，该应用程序的带宽应至少为(1−α)min(Ba,Bg)，其中 α 是容忍级别。 α 越低意味着隔离越严格。 我们用一个例子来演示这个定义是如何工作的：假设攻击者和受害者配置为共享相同的 100 Gbps 网络，并且我们将 α 设置为 25%。 如果受害者单独运行时可以达到 60 Gbps，那么在攻击者的工作负载下，它应该能够至少达到 (1 − 25%)min(60,50) = 37.5 Gbps。 如果受害者单独运行时只能达到10 Gbps，则其消耗的带宽不应小于(1−25%)min(10,50) = 7.5 Gbps。 在实践中，我们发现现有的商用RNIC性能隔离方案都是带宽保证或者可以转化为带宽保证。 我们在第 5 节中使用此定义来表示性能隔离违规，并将 α 设置为 25%。

5 Evaluation

我们使用 NVIDIA 测试台来评估现有的 RDMA 性能隔离解决方案。 测试台中有两台服务器，每台服务器均配备一个 100 Gbps NVIDIA ConnectX5 RNIC。 服务器配备Intel Xeon Gold 5215 CPU，RNIC通过PCIe 3.0 x16连接到服务器。 RNIC 连接到 100 Gbps NVIDIA 交换机。 我们使用Ubuntu 20.04，内核版本是5.11。 对于 NVIDIA NIC，内核驱动程序和动词库均来自 5.4-OFED。 固件版本为16.31.1014。 我们还在 NVIDIA ConnectX-6 RNIC 上进行了所有实验，结果相似。

我们评估了 RNIC 供应商提供的 3 种不同的隔离解决方案和之前的工作：(1) NVIDIA 单独的硬件流量类别 (HW TC)。 云运营商可以为不同的租户设置单独的TC，将RNIC带宽和数据包缓冲区分开[47]以强制执行性能隔离。 现代 RNIC 通常只有 8 个流量类别。 这意味着当我们想要在物理服务器中托管超过 8 个租户时，我们无法使用 HW TC。 (2) NVIDIA SR-IOV。 虽然SR-IOV技术是为硬件虚拟化而设计的，但它为不同的租户提供了独立的虚拟功能和一些独立的资源，实际上实现了一定程度的性能隔离[45]。 (3) Justitia，一种基于软件的性能隔离解决方案[71]。 Justitia 在 RDMA 用户空间库中实现数据verbs速率限制和节奏，以强制执行性能隔离。 这意味着 Justitia 没有安全性：恶意应用程序可以轻松绕过用户空间库。 尽管Justitia的软件架构并不针对多租户公共云环境，但我们仍然使用Husky来评估其隔离策略（例如其基于令牌的算法）的效果。 我们还评估了上述解决方案的所有可能组合。 不幸的是，虽然我们有一个带有 Chelsio T62100-LP-CR 和 Intel E810 NIC 的测试台，但我们没有启用它们基于硬件的隔离机制。 Justitia 也不支持 Chelsio 或 Intel 驱动程序。 因此，我们无法对 Chelsio 或 Intel NIC 进行相同的评估。

5.2 Impact for Real Applications

接下来，我们在更大的测试台上进行实验，以研究在使用最先进的性能隔离解决方案时，微架构资源耗尽如何影响实际应用程序工作负载。 我们在基于 RDMA 的 MPI 实现 [55] 和基于 eRPC 的 Masstree（键值存储）[27, 40] 上使用 allreduce 工作负载 [54] 作为两个真正的受害者应用程序。 我们的测试平台由四台物理服务器组成。 每台服务器均配备一个 100 Gbps NVIDIA ConnectX-5 RNIC。 其他设置与§5.1相同。 受害应用程序在所有四台服务器上运行其虚拟机。 攻击者租户控制两个虚拟机，每个虚拟机位于不同的服务器上。 我们以这种方式设置测试床来模拟真实的多租户环境，因为攻击者可能没有与所有受害者的虚拟机共置的虚拟机。 然而，我们的结果表明，受害者虚拟机子集中违反性能隔离已经足以大幅降低实际分布式应用程序的整体端到端性能。

对于保护机制，我们启用 SR-IOV + HW TC 或 Justitia + HW TC 来为集体通信应用程序提供隔离。 对于基于 eRPC 的 Masstree，我们仅启用 SR-IOV + HW TC。 这是因为 Justitia 仅支持最新 NVIDIA 驱动程序上的高性能 RDMA WRITE，但基于 eRPC 的 Masstree 利用 UD SEND/RECV 进行通信。

我们使用 Husky 测试套件中的四种类型的攻击者来演示我们的结果：（1）BW 攻击是基线。 我们使用标准 Perftest [56] ib_write_bw 来设置一个需要带宽的应用程序。 它使用 16 个 RC QP，每个 QP 保留 128 个未完成的 1 MB 写入请求，以使链路带宽饱和（启用速率限制器时消耗约 50 Gbps）。 BW 攻击不针对任何微架构资源。 (2) PCIe攻击耗尽PCIe TX带宽。 它在 6 个内核上运行 36 个 RC QP，并保留 128 个未完成的 257 B 写入请求。 它还消耗近 50 Gbps 链路带宽（低于 20 Mrps），但会导致超过 73 Gbps PCIe TX 带宽消耗。 这使得受害者只剩下大约 50 Gbps 的可用 PCIe TX 带宽（即小于 50 Gbps 的可用网络带宽）。 (3)缓存攻击耗尽RNIC缓存。 它在 6 个内核上运行 1536 个 RC QP，使用 12288 个 MR，每个 QP 仅保留一个 256 B 未完成请求。 该攻击者会造成严重的缓存未命中，并且仅使用低于 7 Gbps 的链路带宽（即 3 Mrps）。 (4) PU攻击使RNIC PU暂停。 它在单核上运行 1 UC QP，并保留 128 个未完成的 SEND/RECV 请求。 它的接收方不会发出任何接收请求，因此RNIC 必须处理许多接收未就绪异常。 它消耗低于 0.5 Gbps 和低于 0.5 Mrps。

我们从实际应用结果中得到了一些高层次的结论。

要点#1：以微架构资源为目标很容易违反性能隔离。 如果我们把RNIC当作一个黑匣子，那么打破性能隔离是相当困难的。 BW攻击针对的是带宽资源，我们观察到现有的所有解决方案都提供了良好的保护。 然而，一旦我们了解了有关 RNIC 如何工作的更多细节（例如，潜在的微架构资源），打破隔离就变得很简单。 我们的攻击非常有效。 例如，缓存攻击仅需要 7 Gbps 和 3 Mrps。 PU 攻击以更少的带宽和请求率阻止受害者。 请注意，这些攻击仅针对公开披露的微架构组件。

要点#2：应用程序对资源争用的敏感度是不同的。 即使对于相同的攻击，应用程序的端到端性能下降也可能有很大不同。 allreduce 应用程序对缓存耗尽更敏感，而 Masstree 更容易受到 PCIe 耗尽的影响。

要点#3：分布式应用程序需要在每台服务器上进行性能隔离。 对于这两个应用程序，攻击者都只有两台虚拟机，但为什么即使应用程序跨四台机器运行，应用程序级性能也会大幅下降？ 许多现代分布式系统的性能通常受到系统中一些最慢的工作线程的限制。 例如，在allreduce中，每次迭代都需要所有worker的同步。 因此，我们对一两个工人的攻击可能会减慢整个 allreduce 过程。

5.3 Analysis for Existing Solutions

我们的评估表明，所有现有方法都无法提供 RDMA 性能隔离。我们现在分析这些解决方案的基本限制以及我们可能实现的一些潜在改进。

SR-IOV 和独立的 HW TC。 这些基于硬件的解决方案已经提供了一些硬件资源隔离（例如，硬件队列和 NIC 上的数据包缓冲区）。 理论上，RNIC 供应商应该能够将更多硬件隔离功能集成到这些解决方案中。 例如，为不同的VF静态分离NIC PU或分区NIC缓存可以帮助为SR-IOV构建更好的隔离机制。 然而，这些硬件修改是不平凡的，并且很难应用于现有硬件。 RNIC供应商通常将这些新功能与他们的新硬件产品一起发布。 因此，云供应商无法在现有硬件中使用这些功能。

Justitia。 Justitia 提供修改后的用户空间库，并使用发送者准入控制来强制所有租户公平共享带宽和执行速率。 Justitia 不适用于多租户云，因为它的策略不可强制执行：恶意应用程序可以轻松绕过修改后的用户库。 抛开安全方面不谈，值得一问的是，像 Justitia 这样的纯软件解决方案在理论上是否可以支持 RDMA 性能隔离。

我们对这个问题没有直接的答案，但我们相信这是一个有趣的未来研究方向。 我们认为，由于以下原因，这可能相当困难。 首先，在没有硬件支持的情况下，跟踪和控制租户占用了多少缓存是一项挑战。 其次，建立verbs的定量资源消耗模型具有挑战性。 最后，错误处理被深度集成到 RDMA NIC 硬件中，并且对软件来说是不透明的。

6 Guidelines

不幸的是，我们的结果表明，现有的 RNIC 性能隔离解决方案都不够。 我们根据我们的主要发现分析了现有隔离解决方案的失败，并为未来潜在的 RDMA 性能隔离工作提出了一些设计指南。 这些指南也可能有助于 RDMA 应用程序开发人员在多租户环境下编写更好的 RDMA 应用程序。

需要隔离的硬件支持。 像 Justitia [71] 这样的软件方法有一个常见的问题。 它们仅监控架构级别的指标，例如延迟、带宽和请求率。 它们无法检测微架构资源（例如缓存）中的争用，更不用说管理和公平共享这些资源了。 我们相信，未来的性能隔离解决方案将必须利用硬件支持，类似于现代虚拟机管理程序如何使用英特尔资源导向技术（RDT）来监控和管理对最后一级缓存和内存的访问。 NVIDIA RNIC 公开了几个有用的硬件计数器，但它们仍然不够。 例如，我们只能观察缓存未命中，但无法管理缓存访问或为不同租户拆分缓存。

需要一个间接层。 RDMA 意味着数据verbs的内核旁路。 这可实现低延迟并减少 CPU 开销。 那么应该在哪里强制执行性能隔离呢？ 我们相信，未来的性能隔离解决方案将需要 NIC 或软件中的间接层。 在用户态 RDMA 库（如 Justitia）中设置执行点是行不通的，因为它缺乏安全性。 相反，软件间接可以采用类似微内核的设计，其中一组核心在单独的保护域中运行隔离逻辑[43]。 RDMA 性能隔离应该在这样一个同时接管控制verbs和数据verbs的中央控制器中强制执行。

RDMA 应用程序的程序员、编译器和库支持。 未来的性能隔离解决方案发明后，应用程序可能也需要修改。 如果未来的性能隔离解决方案需要对微架构资源进行严格划分，则意味着每个应用程序可使用的微架构资源有限，并可能导致性能大幅下降。 应用程序使用的微架构资源量也可能有所不同（取决于同一服务器或其他配置上有多少其他租户）。 构建高性能 RDMA 应用程序需要程序员、编译器和应用程序库付出额外的努力才能有效地使用这些有限的资源。 对于 CPU 缓存，这些努力发生在二十年前的学术界 [35,36,42]。

7 Discussion

RDMA 性能隔离损坏的影响。 我们的评估表明，恶意租户可能会导致其他租户的性能急剧下降甚至陷入困境。 此外，损坏的性能隔离会暴露恶意用户进行旁路攻击的漏洞。 由于租户可以影响同一主机上其他人的性能，因此它可以设置侧通道来泄漏受害节点的访问模式或通过影响主机的性能来传递信息[65]。 因此，RDMA 性能隔离是安全 RDMA 公共云的一项关键功能。

如今云供应商应使用哪种 RDMA 性能隔离解决方案？ 一个好消息是，据我们所知，目前没有任何云供应商使用商用 RNIC 来为支持 RDMA 的虚拟机提供分区主机资源。 要租用支持 RDMA 的虚拟机，客户必须租用整个物理机。 这意味着目前我们根本不需要RNIC性能隔离解决方案，因为RNIC只运行单个租户的流量。 为了推进 RNIC 的多租户使用，我们认为性能隔离仍然是一个主要障碍，在成熟的性能隔离解决方案准备就绪（至少可以通过我们的测试套件）之前，不应启用多租户。

对其他内核绕过主机网络架构的通用性。 我们的测试套件设计基于 verbs 接口，该接口是 RDMA 特定的。 然而，我们相信我们的方法应该可以推广到其他内核旁路架构（例如 DPDK [13]、1RMA [64]）中发现违反性能隔离的情况，因为这些实现通常需要在设计的 DMA 部分中使用类似 RDMA 的机制。 当今的行业趋势是将功能卸载到硬件加速器。 例如，RDMA 将拥塞控制和可靠的消息传递转移到硬件中。 硬件中的微架构资源对于提供这些卸载的功能至关重要。 关注这些微架构资源以实现性能隔离将变得越来越重要。

8 Related Work

RNIC 中的微架构资源。 RNIC 微架构资源的存在在网络社区中是众所周知的，并且许多研究集中在如何设计 RDMA 应用程序以规避由于这些资源而导致的某些 RNIC 性能异常。 例如，HERD [28]、FaSST [30] 和 eRPC [27] 避免使用 RDMA 可靠连接来减轻 QP 上下文缓存缺失，以获得更好的可扩展性。 ScaleRPC [5] 和 Flock [50] 以分时方式复用可靠连接，以缓解可扩展性问题。 Kalia等人。 [29]研究RNIC的PCIe行为并提供编写高效 RDMA 程序的指南。 不幸的是，这些工作只关注优化应用程序以充分利用 RNIC 中的有限资源。 然而，公有云提供商无法控制第三方租户的应用程序。 Collie [33]对RDMA性能异常进行了系统的搜索，异常大多是由于微架构资源超额认购造成的。 不过，由于Collie只专注于第一方流量，因此它只是在正常操作的基础上构建了一个搜索空间。 因此，它只考虑正常的数据verbs，而无法发现与其他类型行为相关的发现。

例如，第 3 节中的关键发现 #1、#2 和 #3 基本上不包含在 Collie 的搜索空间中，因为 Collie 没有考虑控制verbs、错误处理和昂贵的原子动词。 总的来说，之前的作品更多地从以下角度出发：

应用程序开发人员。 我们的工作是从公共云提供商的角度来看的一个补充方面：这些微架构资源如何影响性能隔离。 这要求我们具有微架构资源意识，并查看所有类型的 RDMA 行为，包括控制verbs和错误处理，因为我们需要处理行为不当甚至恶意的租户。

其他 NIC 性能隔离解决方案。 PicNIC [34] 为 NIC 上的数据包处理和带宽提供隔离。 这使得延迟受限的工作负载不会受到带宽受限的工作负载的影响。 FairNIC [15] 隔离基于 SoC 的 SmartNIC 中的资源。 与它们相比，我们的工作重点是NIC上的RDMA相关资源。

其他上下文中的性能隔离。 性能隔离问题不仅限于网卡。 其他服务器硬件组件也存在这个问题，并且他们已经有了相应的解决方案。 CPU 缓存 [11, 20] 和内存带宽 [22] 存在多种分区技术。 网络结构中的网络带宽也是隔离 [1, 3, 4, 6, 16, 24, 25, 37, 58–60, 62, 68] 以及交换机处理管道的重要资源 [67]。

9 Conclusion

RDMA 是一种很有前景的网络技术，可在数据中心网络中实现低延迟和高 CPU 效率。 为了在多租户环境中启用RDMA，性能隔离是一个重要的属性，并且由于微架构资源（例如，RNIC缓存、处理单元）的存在，RDMA NIC（RNIC）带来了新的挑战。 我们提出了一个关于不同 RDMA 操作如何使用这些资源的 RNIC 操作模型。 使用此模型，我们创建了 Husky，这是第一个评估 RNIC 性能隔离解决方案的测试套件。 我们的结果表明，现有的 RNIC 性能隔离解决方案都无法针对试图耗尽这些微架构资源的工作负载提供足够的隔离。 我们的研究结果得到了最大的 RDMA NIC 供应商之一的认可和复制。 我们相信构建可用的RNIC性能隔离解决方案将是一场持久战。
