二、内容分发

CDN：Content Delivery Network（内容分发网络）

  • 存什么？
    • 视频：10-20%的视频产生80%的访问
  • 存不下怎么办？

2.1访问的局部性

  • 80-20准则：20%的内容吸引80%的访问
  • zipf定律---局部性判断： 排名第i位的内容访问次数 \(y_{i}=1/i^{s}\)
    • Log-log坐标中，是一条直线

  • Web Cache：在靠近用户网络的地方缓存流行内容，后续请求不需要再到原服务器

    运营商采用（如电信），节省流量

  • CDN：把内容拷贝到不同地域的多台服务器，减轻服务器负载， 提升用户感知的质量

    内容提供商（如PPTV），提升服务质量

2.2如何实现

CDN：主动发布内容

核心问题：选择哪个服务器

  • 负载轻的服务器
  • 性能高的服务器
    • 较高的处理能力，带宽等
  • 距离客户端近的服务器
    • 距离可以是物理距离（如一个城市），也可 以用RTT来衡量
  • 成本（包括带宽、能耗等）较低的服务器

2.3 EDNS

假设不使用EDNS，当客户端使用Google：8:8:8:8 等公共DNS解析服务器进行内容访问时，CDN有可能会给用户分配的CDN服务器离客户端距离较远，原因是什么？ EDNS为什么能协助解决这个问题？ Google的8:8:8:8使用Anycast技术也能在一定程度上缓解该问题。请问原理是什么？

假设不使用EDNS，当客户端使用Google的8.8.8.8等公共DNS解析服务器进行内容访问时，可能会出现CDN服务器距离客户端较远的情况，原因主要涉及DNS解析和内容分发网络（CDN）的工作原理。

不使用EDNS时的问题：

  1. DNS解析限制： 原始的DNS协议在设计时限制了每个UDP数据包的大小为512字节。这种限制导致在DNS响应中无法包含足够的信息，特别是关于客户端位置和网络拓扑的信息。因此，CDN服务无法根据客户端的地理位置信息做出最佳的服务器选择。
  2. 缺乏客户端信息： 由于缺乏足够的数据，CDN服务无法确定客户端的地理位置或网络接入点，从而无法将用户重定向到最近的CDN服务器。

EDNS如何解决这个问题：

  1. 支持更大的DNS消息： EDNS允许DNS消息的数据部分更大，使得DNS响应可以包含更多的信息。这样，DNS响应可以携带有关客户端位置、网络拓扑和其他相关信息的扩展数据。
  2. 客户端子网信息： EDNS0（EDNS版本0）中引入了Client Subnet扩展，允许DNS客户端将其子网信息包含在DNS请求中。这使得DNS解析服务器能够获取有关客户端所在子网的信息。
  3. 更智能的CDN选择： 基于EDNS提供的额外信息，CDN服务能够更智能地选择最适合客户端的服务器。通过了解客户端的位置，CDN可以将用户路由到距离最近的服务器，从而提高内容传递的效率。

Google的8.8.8.8使用Anycast的原理：

Google的8.8.8.8采用Anycast技术，这是一种将一个IP地址广播到多个地理分布的服务器的方法。具体原理如下：

  1. 多个服务器： Google在全球范围内部署了多个8.8.8.8 DNS服务器，这些服务器使用相同的IP地址。
  2. 路由选择： 当客户端向8.8.8.8发出DNS查询时，Internet路由系统会选择距离最近的物理服务器。Anycast通过选择网络上最近的路由路径来实现这一点。
  3. 最佳路径： 由于Anycast路由系统的工作方式，用户的DNS查询将被路由到物理网络上最近的可用服务器，因此可以降低响应时间并提高性能。

Anycast技术在一定程度上缓解了DNS解析服务器位置选择的问题，但仍然可能受到DNS协议的限制。EDNS的使用可以进一步优化Anycast的效果，通过提供更多的信息来实现更精准的CDN服务器选择。

三、分层模型

网络通信中需要很多很多协议、功能、组件的协作，如何使用这些协议构建计算机网络？

协议(Protocol)：定义通信参与方如何交互的一种协定和规范

方案1 – 模块化

  • 模块化的协议栈 + 良好定义的模块接口
  • 优点: 独立性强，功能简单，模块易于实现
  • 缺点: 适应性差，难以维护，对系统实现是场灾难

方案2 – 两层结构

将计算机网络分为两层

  • 应用层与物理层，两层的每种类型一对一适配
  • 缺点: 每有一种新的应用，都需要与物理层的每种类型去适配

方案3 – 三层结构

引入中间层，分别与应用层和物理层适配

  • 中间层实现对物理层的抽象，对应用层提供统一接口

  • 极大降低了适配的工作量， \(M*N -> M+N\)

  • 缺点: 引入适配层增加了设计难度 (可忽略不计）

分层模型工作机制：封装和复用

分层模型的改进：现实中，网络环境的复杂性使得分层独立设计难以达到最优性能，跨层(Cross Layer)感知可以弥补这一缺陷

互联网体系结构可自下而上分为链路层，网络层，传输层和应用层，请简述各层主要功能，与代表性协议，以及该分层模型的优缺点。

  1. 链路层
    • 功能：可靠的点对点数据传输，进行帧编码和差错控制
    • 代表协议：
      • 点对点协议
      • 以太网协议
  2. 网络层
    • 功能：进行网络互连、实现端到端传输
    • 代表协议
      • IP协议
      • ICMP协议
      • ARP协议
  3. 传输层
    • 功能：提供可靠的端到端数据传输
    • 代表协议
      • TCP协议
      • UDP协议
  4. 应用层
    • 功能：为用户应用进程提供服务
    • 代表协议
      • DNS
      • HTTP
      • FTP

四、直连网络

4.1 性能指标

  • 带宽(Bandwidth): 单位时间内通过链路所能传输的比特数
  • 延迟(Delay): 消息从网络一端传到另一端所需的时间
    • 传播时延 + 处理延迟 + 排队延迟

4.2 数据帧封装

在一段数据的前后分别添加首部和尾部，就构成了一个帧

首部和尾部的作用是界定帧的范围，不同的数据链路协议有不同的格式定义，以BISYNC为例

如果数据负载中也包含STX或ETX字符如何处理？

引入转义符DLE，将数据负载中所有的STX, ETX, DLE进行转义

4.3差错检测方法

差错检测算法的设计目标：使用最少的冗余比特检测最多的错误

  • 奇偶校验
  • 互联网校验和 (checksum)
  • 循环冗余校验

4.4 可靠传输

可靠传输基本思想

  • 确认(acknowledgment, ACK)：接收方接受数据帧，回复ACK帧
  • 超时(timeout)：发送方在规定时间(timeout)内没收到ACK，重传该数据

4.4.1停等(stop-and-wait)协议

缺点：链路每次只能传输一个数据帧，带宽利用率低

  • 10Gb/s以太网链路，往返时延10us，帧大小1500B
  • 停等协议传输速率：1500Bytes/10us = 1.2Gb/s

4.4.1.1 提升传输速率

提升传输速率的根本方法是增加单位时间传输数据帧的数目

  • 通过窗口大小(window)限制在途传输的数据帧个数
  • 允许多个在途传输(未收到ACK)的数据帧
  • 每个数据帧赋予一个序列号 (Seq)，数据传输和确认都基于Seq

为什么说滑动窗口协议是一种高效的可靠传输机制？

  • 可靠传输：具有数据确认和超时重传机制
  • 高效传输：通过并发提升传输性能
  • 按序传送：接收方将连续的数据交给上层，不连续的数据存放在bufffer中
  • 流控功能：双方通过设定窗口大小表达自己的发送或接受能力

4.5 多路复用技术

  • 频分复用（FDM）
  • 时分复用（TDM）
  • 统计时分复用（STDM）
  • 码分复用（CDM）
  • 载波侦听多路访问（CSMA）

五、交换(Switching)网络

设计目标

  1. 数据只朝向目的节点方向传送（转发, Forward）
  2. 转发规则是网络自己学习生成的，不需要外界参与

两个主要部分

  1. 数据帧转发
  2. 学习节点位置

5.1 数据帧转发

给定一个包含源目的MAC地址的数据帧，如何确定从哪个端口转出？

  • 交换机存储目的MAC地址到端口的映射关系(ForwardingDatabase, FDB)

  • 对于每个数据帧，在FDB中查找目的MAC地址对应的端口号

    • 如果存在对应端口号，从该端口将数据转发（单播）
    • 如果FDB中不存在对应条目，将该数据包从所有端口转发（广播）

  • FDB表中MAC地址通过老化机制(Aging)来更新

问题：FDB条目是如何生成的？

每收一个新的数据帧，记录其源MAC地址和入端口，将该映射关系写入FDB表

5.2 生成树算法

生成树算法举例

  • 节点N1:
    • 一直发送(N1, N1, 0)
  • 节点N2：
    • 发送(N2, N2, 0)
    • 收到N1消息后，更新并向N3发送(N2, N1, 1)
  • 节点N3：
    • 发送(N3, N3, 0)
    • 收到N4的消息(N4, N1, 1)后，更新并向N2发送(N3, N1, 2)
    • 收到N2的消息(N2, N1, 1)后，更新并停止发送

是否可以将交换网络扩充到全球范围？

  • 不可以，存在可扩展性问题、网络接入方式多样性问题
    • FDB表的膨胀问题
    • STP收敛速度问题

6.1地址解析协议(ARP)

IP层抽象屏蔽了下层MAC协议的细节

  • 在链路层只能看见 MAC 帧而看不见 IP 数据报
  • 数据传输时，需要IP地址到MAC地址的映射

ARP：知道下一跳IP地址，查询其MAC地址

节点A如何向局域网内另一节点B发送IP报文？

  • 如果ARP Cache中有对应条目，将该MAC地址作为目的地址发送数据帧
  • 否则，A向局域网内广播ARP请求，询问节点B的IP地址对应的MAC地址
    • B收到该请求后，回复自己的MAC地址
    • A和B都会将对方地址的映射关系写入ARP Cache

注意：==ARP协议只作用于局域网==，两节点IP不在同一网段时，数据报文应先转发到路由器
