2.3 EDNS

假设不使用EDNS，当客户端使用Google：8:8:8:8 等公共DNS解析服务器进行内容访问时，CDN有可能会给用户分配的CDN服务器离客户端距离较远，原因是什么？ EDNS为什么能协助解决这个问题？ Google的8:8:8:8使用Anycast技术也能在一定程度上缓解该问题。请问原理是什么？

假设不使用EDNS，当客户端使用Google的8.8.8.8等公共DNS解析服务器进行内容访问时，可能会出现CDN服务器距离客户端较远的情况，原因主要涉及DNS解析和内容分发网络（CDN）的工作原理。

不使用EDNS时的问题：

  1. DNS解析限制： 原始的DNS协议在设计时限制了每个UDP数据包的大小为512字节。这种限制导致在DNS响应中无法包含足够的信息，特别是关于客户端位置和网络拓扑的信息。因此，CDN服务无法根据客户端的地理位置信息做出最佳的服务器选择。
  2. 缺乏客户端信息： 由于缺乏足够的数据，CDN服务无法确定客户端的地理位置或网络接入点，从而无法将用户重定向到最近的CDN服务器。

EDNS如何解决这个问题：

  1. 支持更大的DNS消息： EDNS允许DNS消息的数据部分更大，使得DNS响应可以包含更多的信息。这样，DNS响应可以携带有关客户端位置、网络拓扑和其他相关信息的扩展数据。
  2. 客户端子网信息： EDNS0（EDNS版本0）中引入了Client Subnet扩展，允许DNS客户端将其子网信息包含在DNS请求中。这使得DNS解析服务器能够获取有关客户端所在子网的信息。
  3. 更智能的CDN选择： 基于EDNS提供的额外信息，CDN服务能够更智能地选择最适合客户端的服务器。通过了解客户端的位置，CDN可以将用户路由到距离最近的服务器，从而提高内容传递的效率。

Google的8.8.8.8使用Anycast的原理：

Google的8.8.8.8采用Anycast技术，这是一种将一个IP地址广播到多个地理分布的服务器的方法。具体原理如下：

  1. 多个服务器： Google在全球范围内部署了多个8.8.8.8 DNS服务器，这些服务器使用相同的IP地址。
  2. 路由选择： 当客户端向8.8.8.8发出DNS查询时，Internet路由系统会选择距离最近的物理服务器。Anycast通过选择网络上最近的路由路径来实现这一点。
  3. 最佳路径： 由于Anycast路由系统的工作方式，用户的DNS查询将被路由到物理网络上最近的可用服务器，因此可以降低响应时间并提高性能。

Anycast技术在一定程度上缓解了DNS解析服务器位置选择的问题，但仍然可能受到DNS协议的限制。EDNS的使用可以进一步优化Anycast的效果，通过提供更多的信息来实现更精准的CDN服务器选择。

三、分层模型

网络通信中需要很多很多协议、功能、组件的协作，如何使用这些协议构建计算机网络？

协议(Protocol)：定义通信参与方如何交互的一种协定和规范

方案1 – 模块化

  • 模块化的协议栈 + 良好定义的模块接口
  • 优点: 独立性强，功能简单，模块易于实现
  • 缺点: 适应性差，难以维护，对系统实现是场灾难

方案2 – 两层结构

将计算机网络分为两层

  • 应用层与物理层，两层的每种类型一对一适配
  • 缺点: 每有一种新的应用，都需要与物理层的每种类型去适配

方案3 – 三层结构

引入中间层，分别与应用层和物理层适配

  • 中间层实现对物理层的抽象，对应用层提供统一接口

  • 极大降低了适配的工作量， \(M*N -> M+N\)

  • 缺点: 引入适配层增加了设计难度 (可忽略不计）

分层模型工作机制：封装和复用

分层模型的改进：现实中，网络环境的复杂性使得分层独立设计难以达到最优性能，跨层(Cross Layer)感知可以弥补这一缺陷

互联网体系结构可自下而上分为链路层，网络层，传输层和应用层，请简述各层主要功能，与代表性协议，以及该分层模型的优缺点。

  1. 链路层
    • 功能：可靠的点对点数据传输，进行帧编码和差错控制
    • 代表协议：
      • 点对点协议
      • 以太网协议
  2. 网络层
    • 功能：进行网络互连、实现端到端传输
    • 代表协议
      • IP协议
      • ICMP协议
      • ARP协议
  3. 传输层
    • 功能：提供可靠的端到端数据传输
    • 代表协议
      • TCP协议
      • UDP协议
  4. 应用层
    • 功能：为用户应用进程提供服务
    • 代表协议
      • DNS
      • HTTP
      • FTP

四、直连网络

4.1 性能指标

  • 带宽(Bandwidth): 单位时间内通过链路所能传输的比特数
  • 延迟(Delay): 消息从网络一端传到另一端所需的时间
    • 传播时延 + 处理延迟 + 排队延迟

4.2 数据帧封装

在一段数据的前后分别添加首部和尾部，就构成了一个帧

首部和尾部的作用是界定帧的范围，不同的数据链路协议有不同的格式定义，以BISYNC为例

如果数据负载中也包含STX或ETX字符如何处理？

引入转义符DLE，将数据负载中所有的STX, ETX, DLE进行转义

4.3差错检测方法

差错检测算法的设计目标：使用最少的冗余比特检测最多的错误

  • 奇偶校验
  • 互联网校验和 (checksum)
  • 循环冗余校验

4.4 可靠传输

可靠传输基本思想

  • 确认(acknowledgment, ACK)：接收方接受数据帧，回复ACK帧
  • 超时(timeout)：发送方在规定时间(timeout)内没收到ACK，重传该数据

4.4.1 停等(stop-and-wait)协议

缺点：链路每次只能传输一个数据帧，带宽利用率低

  • 10Gb/s以太网链路，往返时延10us，帧大小1500B
  • 停等协议传输速率：1500Bytes/10us = 1.2Gb/s

4.4.1.1 提升传输速率

提升传输速率的根本方法是增加单位时间传输数据帧的数目

  • 通过窗口大小(window)限制在途传输的数据帧个数
  • 允许多个在途传输(未收到ACK)的数据帧
  • 每个数据帧赋予一个序列号 (Seq)，数据传输和确认都基于Seq

为什么说滑动窗口协议是一种高效的可靠传输机制？

  • 可靠传输：具有数据确认和超时重传机制
  • 高效传输：通过并发提升传输性能
  • 按序传送：接收方将连续的数据交给上层，不连续的数据存放在bufffer中
  • 流控功能：双方通过设定窗口大小表达自己的发送或接受能力

4.5 多路复用技术

  • 频分复用（FDM）
  • 时分复用（TDM）
  • 统计时分复用（STDM）
  • 码分复用（CDM）
  • 载波侦听多路访问（CSMA）

五、交换(Switching)网络

设计目标

  1. 数据只朝向目的节点方向传送（转发, Forward）
  2. 转发规则是网络自己学习生成的，不需要外界参与

两个主要部分

  1. 数据帧转发
  2. 学习节点位置

5.1 数据帧转发

给定一个包含源目的MAC地址的数据帧，如何确定从哪个端口转出？

  • 交换机存储目的MAC地址到端口的映射关系(ForwardingDatabase, FDB)

  • 对于每个数据帧，在FDB中查找目的MAC地址对应的端口号

    • 如果存在对应端口号，从该端口将数据转发（单播）
    • 如果FDB中不存在对应条目，将该数据包从所有端口转发（广播）

  • FDB表中MAC地址通过老化机制(Aging)来更新

问题：FDB条目是如何生成的？

每收一个新的数据帧，记录其源MAC地址和入端口，将该映射关系写入FDB表

5.2 生成树算法

生成树算法举例

  • 节点N1:
    • 一直发送(N1, N1, 0)
  • 节点N2：
    • 发送(N2, N2, 0)
    • 收到N1消息后，更新并向N3发送(N2, N1, 1)
  • 节点N3：
    • 发送(N3, N3, 0)
    • 收到N4的消息(N4, N1, 1)后，更新并向N2发送(N3, N1, 2)
    • 收到N2的消息(N2, N1, 1)后，更新并停止发送

6.1地址解析协议(ARP)

IP层抽象屏蔽了下层MAC协议的细节

  • 在链路层只能看见 MAC 帧而看不见 IP 数据报
  • 数据传输时，需要IP地址到MAC地址的映射

ARP：知道下一跳IP地址，查询其MAC地址，每个路由器/主机中都有一个ARP缓存（Cache

节点A如何向局域网内另一节点B发送IP报文？

  • 如果ARP Cache中有对应条目，将该MAC地址作为目的地址发送数据帧
  • 否则，A向局域网内广播ARP请求，询问节点B的IP地址对应的MAC地址
    • B收到该请求后，回复自己的MAC地址
    • A和B都会将对方地址的映射关系写入ARP Cache

注意：ARP协议只作用于局域网，两节点IP不在同一网段时，数据报文应先转发到路由器

6.2 Buffer Bloat问题

BufferBloat是指数据包在缓冲区中存留时间过长引起的延迟过大问题

6.2.1 产生原因

  1. 设备的队列设置过大： 当网络设备的队列设置得过大时，会导致数据包在队列中积压，增加了网络的延迟。这是因为大队列会导致数据包在队列中停留的时间过长，而不是被及时传输。
  2. 难以精确计算需要多大的队列： 确定合适的队列大小是一个复杂的问题，因为需要考虑网络流量的变化、设备之间的传输速率不匹配等因素。缺乏精确计算导致可能设置过大的队列，从而引发BufferBloat问题。
  3. 在成本允许的前提下，缓冲区越大越好： 由于缓冲区的大小直接关系到设备的成本和性能，一些设备可能会选择在成本允许的前提下设置尽可能大的缓冲区。然而，这可能导致过度的缓冲，从而增加延迟。

在以下情况下更容易发生

  1. 网络负载较重的时间段：不会一直持续
  2. 边缘网络： 这是因为边缘网络可能面临更多的流量变化和传输速率不匹配的情况，而且对网络性能的影响更为敏感。
  3. 3G/4G网络：在移动网络（如3G和4G）中，运营商为了提升服务质量（Quality of Service，QoS）可能会部署大量的队列。

6.2.2 解决

  1. 减小队列大小
    • 减小队列大小可以降低数据包在队列中的时间
    • 但是，小队列难以容忍突发流量
  2. 改进传输控制策略
    • 考虑延迟变化，不将丢包作为TCP传输控制的唯一信号
  3. 改进队列管理策略
    • 在队列满之前就主动（概率性的）丢包：RED (Random Early Detection)
    • 以延迟作为队列管理的信号：CoDel (Controlled Delay)

7.2 方法比较

距离向量方法：

  1. 没有全网统一的度量指标： 在距离向量方法中，不同的Autonomous Systems（AS）可能使用不同的度量标准，因为每个AS都有自己的路由策略。
  2. Bellman-Ford算法收敛速度较慢： 距离向量方法中使用的Bellman-Ford算法在某些情况下可能导致收敛速度较慢，而且存在Count-to-Infinity问题。
  3. 易于部署： 距离向量方法相对简单，易于部署和管理，特别适用于较小规模的网络。

链路状态方法：

  1. 度量指标不一致可能导致环路： 链路状态方法中，每个系统使用的度量指标可能不一致，这可能导致环路的出现。不同系统使用的度量标准不一致可能会对网络造成负面影响。
  2. 链路状态数据库太大： 链路状态方法要求每个节点维护一个全局的链路状态数据库，对于大规模网络，这可能导致数据库庞大，占用大量内存。
  3. 暴露路由策略给其他AS： 链路状态方法可能会暴露自己的路由策略给其他AS，这可能是不希望发生的，尤其是在有安全和隐私方面的考虑时。

综合考虑：

  • 对于较小规模的网络，距离向量方法可能更简单、易于管理，并且收敛速度较慢的问题相对较小。
  • 对于大规模网络，链路状态方法可能更灵活，但需要处理一些与度量标准和数据库大小相关的挑战

7.3 BGP

BGP（Border Gateway Protocol，边界网关协议）是一种路径向量协议，用于在不同的自治系统（AS）之间交换路由信息。以下是关于BGP的一些关键特点：

  1. 路径向量（带路径的距离向量）：
    • BGP是一种路径向量协议，与距离向量协议相似，但每个路由更新中携带整条路径信息。这使得BGP能够维护和传递更详细的路由信息，包括经过的AS的路径。
  2. 检查环路：
    • BGP通过检查路径信息来防止环路的发生。当AS收到路由更新消息后，它会检查自身是否在对应路径中。如果存在环路，AS会丢弃该更新消息，避免形成循环路径。
  3. 优点：
    • 度量指标不需要全局统一：BGP不要求所有的AS使用相同的度量标准，每个AS可以根据自身的策略和需求来选择路径。
    • AS选择路径，协议检查避免可能的环路：BGP允许AS根据自身策略选择最优的路径，并通过协议内置的环路检测机制来避免循环。
  4. BGP协议运行在自治系统AS之间：
    • BGP并不像内部路由协议（IGP）一样在同一AS内生成路由。它的主要作用是在不同的自治系统之间传播路由信息，以实现全球互联的互联网。
  5. 控制路由的传播和选择最优的路由：
    • BGP的主要目的是控制路由的传播，使得不同的AS可以根据自身的政策选择最优的路由。BGP不直接生成路由，而是从IGP中引入路由或从其他BGP对端学习路由。

八、网络传输机制

UDP (User DatagramProtocol)：最简单的传输层协议，根据端口号进行多路复用

为什么需要UDP协议？

  • 不需要建立连接、不需要维护状态
    • 减少启动延迟，例如DNS
  • 不需要可靠传输
    • 例如视频直播等应用
  • 作为最基本的传输层协议，上层应用可实现更多功能、按需定制
    • 例如UDT、QUIC等

TCP (Transport Control Protocol)，应用最广泛的传输层协议，占据了目前互联网的90%以上流量

  • 多路复用
    • Src Port, Dst Port
  • 连接管理
    • Flags + Seq + Ack
  • 可靠传输
    • Seq + Ack + Options
  • 流控
    • Recv Window + Options
  • 拥塞控制

传输机制本质就是向上适配应用向下适配网络

8.1 网络拥塞(Congestion)

当资源需求超过网络容量时，产生问题

  • 每条流不知道当前网络资源分配情况
  • 每条流也不知道其它(竞争)流的存在

后果：丢包率升高、往返时间增大、甚至网络崩溃(Network Collapse)

9.3 性能优化

9.3.1 初始窗口

  • 初始窗口可能会限制效率
  • 窗口和在slow start内的时间关系： \(T ≈ log_{2}(S/init\_rwnd + 1) * RTT\)
  • 短流会在slow start阶段结束，不能充分利⽤带宽 – Outstanding包少的时候(<4)，触发不了快速重传，因为 收集不齐3个重复ACK

通过增大 TCP 初始窗口，可以提高短流的传输效率，特别是在 slow start 阶段，以更好地利用可用的带宽。

9.3.2 超时重传

  • RTO（Retransmission Timeout）是超时重传的时间，它可能是往返时间（RTT）的几个数量级以上。在RTO时间内，不能传输数据，而超时重传会导致进入 slow start 阶段。
  • 对于短流来说，RTO的影响很大
  • 优化RTO问题 - TLP：
    • 在出现超时时发送一个称为 TLP Probe 的数据包，以探测是否有尾部数据包丢失。
    • 目标是在不使用超时重传的情况下，更快地恢复丢失的数据包。通过采用这种更主动的方式，TLP 可以提高对丢失数据包的敏感性，而无需等待整个 RTO。
  • 优化RTO问题-SRTO：
    • SRTO: Smart-RTO

9.5 存在问题

  1. 3次握手：
    • TCP 建立连接时需要进行3次握手，即使只传输一个数据包的情况下也要执行完整的握手过程。这可能引入一定的延迟，特别是对于短连接的情况。
  2. HTTPS 协商时间：
    • 在使用HTTPS协议时，协商过程可能需要较长的时间，这是由于需要进行密钥协商等安全性操作。这可能导致一些连接的启动时间较长。
  3. 内核态的限制：
    • TCP 的一些优化可能需要在操作系统的内核态进行，而不容易与应用程序结合做定制优化。这可能限制了一些性能优化的实施。
  4. 与IP地址绑定：
    • TCP 连接通常与IP地址绑定，这在多路径或移动场景下可能会带来一些不便，特别是在移动设备切换网络时。
  5. HoL问题（头部阻塞）：
    • TCP 要求数据包按照顺序到达，这可能导致 Head-of-Line (HoL) 阻塞问题。如果某个数据包丢失或延迟，后续的数据包将受到影响。
  6. 扩展选项问题：
    • 一些 TCP 的扩展选项，如用于支持多路径传输的MPTCP等功能，经常被防火墙等设备扔掉。这可能导致这些功能在网络中难以大规模部署和应用。
